Node colours — certainty level

Green > 80% — fact / proven
Blue 60–80% — consensus / probable
Orange 40–60% — working hypothesis
Red < 40% — speculation / unknown
Rule: Colour is a property of certainty, shape is a property of role. A red QUESTION is an uncertain hypothesis. A green BLACKBOX is a verified unknown. Combinations are intentional.

Core edges (required)

CAUSES
grey / neutral — main spine
A directly produces B. Primary causal edge. Default choice for every connection.
ENABLES / REQUIRES
green — existence condition
Without A, B cannot occur. Does not say A directly causes B — only removes the barrier.
BLOCKS
red — interruption —|
A prevents B. A TASK or DECISION intervention typically blocks a negative EVENT.
REDUCES
blue — constructive intervention
TASK or DECISION actively addresses B. The opposite of BLOCKS — repairs instead of prevents.

Extended edges (intensity and mechanism)

AMPLIFIES / REDUCES
grey with symbol ▲ / ▼
A modulates the intensity of B, but B would exist without A. Typical for feedback loops.
TRIGGERS
grey dashed — trigger
A is the condition for EVENT B. Distinction from CAUSES: a state triggers an event, not another state.
THRESHOLD CROSSING
orange — threshold crossing
Use when the causal effect depends on reaching a specific value (OOM Kill, critical stress).
STATE→EVENT matrix: If you see STATE → CAUSES → STATE, ask: "What changed?" — an EVENT node is missing in the middle. STATE → TRIGGERS → EVENT → CAUSES → STATE is the correct form.

RCDE workflow — 4 steps

4
EFFECT — define the impact (start here)
"What happened, or what do we want to achieve?"
Place GOAL or result STATE to the bottom right. If the impact has not yet occurred, it is a GOAL (triangle). If it has, STATE or EVENT.
1
ROOT CAUSE — find the root
"Where does the problem originate? What triggered it?"
Place STATE or EVENT to the top left. If the root is unknown, insert a QUESTION (star) — it is a legitimate node, not an error. Ask "Why?" at most 5 times.
2
CAUSAL PATHWAY — connect the path
"How does the root lead to the impact?"
Connect Root and Effect nodes via STATE / EVENT. Every STATE→STATE transition needs an EVENT in the middle. Max 4–5 nodes between root and gate — otherwise insert a GATE.
3
DECISION POINT — insert intervention
"Where can the chain be interrupted or redirected?"
Insert DECISION or TASK at the point where influence on the chain is possible. TASK + BLOCKS = prevention. TASK + REDUCES = repair. DECISION always branches into ≥ 2 outputs with different impacts.

Pre-completion checklist

☐ Every STATE→STATE transition has an EVENT node
☐ Every DECISION has ≥ 2 branches with effects
☐ Every GATE has a defined condition (AND/OR/XOR)
☐ Every BLACKBOX has a certainty colour
☐ GOAL node closes the right end of the diagram
☐ QUESTION nodes are marked as research tasks
☐ ASSET nodes have an ENABLES link to the following node

Structural rules

Transition rule: state → event → state
Two STATE nodes cannot be directly connected via CAUSES without an intermediate EVENT. STATE "Memory Leak" → EVENT "OOM Kill" → STATE "Outage" — correct. STATE → STATE — error.
Depth rule: max 4 nodes without GATE
If a causal path contains more than 4 nodes without branching or a GATE, insert a GATE with a condition. Long straight chains hide latent branching.
Completeness rule: every branch has an Effect
Every output branch of a DECISION or GATE must end at STATE, GOAL, or a feedback loop. Dead-end branches = incomplete modelling.
BLACKBOX rule: write the symptom, not the cause
BLACKBOX contains a description of the effect, not an assumption about the cause. Correct: "Leak did not appear in test". Incorrect: "Probably a broken mock".
ASSET rule: condition, not cause
Use an ASSET (cylinder) node only when its absence stops the chain. Connect it with ENABLES, never CAUSES.
Colour consistency rule
Node colour relates to the certainty of its content, not the node type. If STATE "Chronic stress" is only a hypothesis (60%), it is blue — regardless of shape.

Temporal orientation

time →
Root on the left, Effect on the right. DECISION and TASK are placed at the point in the timeline where the intervention actually occurs — not always at the end. Feedback loops may lead back to the left.

Common mistakes

✕
State–State without event
"Chronic stress → Collapse" — missing EVENT. Correct: "Chronic stress → reduces → Sleep quality → triggers → Collapse".
✕
Decision without branching
A DECISION node with one output edge is not a decision — it's a TASK. If there aren't at least two choices with different impacts, change the shape.
✕
Everything green
If all nodes are green (> 80%), the diagram does not make epistemics visible. Every model has uncertain areas — find them and colour them orange or red.
✕
CONCEPT as a substitute for STATE
"Expansion" is a CONCEPT, but "Access to 10,000 new customers" is a STATE. If it can be made concrete, use STATE. Leave CONCEPT for unmeasurable abstractions.
✕
Chain without GATE after 5+ nodes
Too straight a chain = skipped branching or conditions. Insert a GATE with the question "Does this always happen?" — if not, there is a condition that belongs in the model.
✕
Mixing CAUSES and ENABLES
ASSET "Budget" → CAUSES → "Expansion" is an error. Correct: ASSET "Budget" → ENABLES → DECISION "Choose region". Without a budget you cannot decide, but the budget alone will not cause expansion.
